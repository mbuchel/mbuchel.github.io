Machine-code rewrite · H100

Rewriting FlashAttention's machine code

The same attention, fewer instructions — faster on Hopper.

9.6155ms → 7.6539ms · 1.26–1.29× faster (long context)
Part of an in-review research project

sass2mlir is the subject of a paper we’re preparing for peer review. This post is a high-level walkthrough — by design it holds back the implementation details, the full conversion-pass library, the SASS encoding internals, and the broader validation numbers while the work is under review, and nothing here is open-sourced yet. The paper also covers an online, on-the-fly conversion path — rewriting a kernel’s machine code as it loads, not just offline — which we only gesture at here. Treat this as the intuition, not the full method.

The problem

The official kernel under-uses the H100

FlashAttention’s official forward kernel was tuned for an earlier GPU generation’s habits. On an H100 it asks for 255 registers per thread — the hardware maximum — which caps occupancy at 12.5% and forces 85 register spills to local memory. It moves data with cp.async + ldsm and multiplies with per-warp HMMA tensor-core ops. All correct — and all leaving Hopper’s newest hardware idle.

Show the SASS detail
  • Registers/thread: 255 (max) → block-limited to 2 → 12.5% occupancy
  • Register spills (STL/LDL pairs): 85
  • Tensor MMA: 512 per-warp HMMA instructions
  • Load path: cp.async (48 LDGSTS) + 144 LDSM
  • Launch grid: 256,2,4

And rebuilding from source doesn’t fix it. This kernel was compiled from source, on the H100, targeting Hopper (sm_90) — and the compiler still emitted the register-bound, pre-Hopper code above: per-warp HMMA instead of warpgroup HGMMA, cp.async instead of TMA, 85 spills, 12.5% occupancy. The hardware supports the faster path — the Triton compiler reaches it on the very same GPU — but the official source plus the nvcc/CUTLASS toolchain leaves those wins on the table. We tracked down why: it was compiled for sm_90, but the chip is sm_90a. Hopper’s headline instructions — warpgroup HGMMA (wgmma) and TMA — are gated behind the architecture-specific sm_90a target. Built for plain sm_90, ptxas simply can’t emit them and falls back to Ampere-class HMMA + cp.async; Triton targets sm_90a, so it reaches the fast path on the very same H100. That is the gap a tool like sass2mlir fills: it works below the source compiler, on the SASS it actually emitted — lift it to MLIR, apply the optimizations the toolchain missed, re-emit and reinject — so you can re-optimize for the exact GPU you have without waiting on upstream to change the source.

The insight

The same math, a newer instruction set

Compare the official SASS against a Triton-compiled kernel doing the identical attention. The optimized version replaces per-warp tensor ops with warpgroup async MMA and the manual copy pipeline with TMA + hardware barriers:

- HMMA.16816.F32.F16  R8, R24, R40, R8   // per-warp MMA, operands in regs
+ HGMMA.64x64x16.F32.F16   // warpgroup async MMA, smem descriptors

- LDGSTS.E.128  [R12], [R4]   // cp.async global→shared
+ UTMALDG.2D  [desc], [R2]    // TMA bulk load + mbarrier

It distills to six transforms:

  1. HMMA (per-warp tensor MMA) → HGMMA (warpgroup async MMA)
  2. cp.async + ldsm → TMA (UTMALDG) + mbarrier (SYNCS)
  3. scalar work → uniform-register path (URn, ELECT, VOTEU)
  4. monolithic warp → producer/consumer warp specialization
  5. FFMA.FTZ → FFMA (drop FTZ)
  6. register spills (STL/LDL) → none (the uniform path frees the register budget)

The keeper

sass2mlir: rewrite the machine code, keep the ABI

The real artifact is a compiler round trip: disassemble the official kernel, lift it into MLIR, apply conversion passes that implement the six rules, re-encode to SASS, and reinject while preserving the exact kernel ABI — so it drops into the existing binary with no host changes. Unlike a one-off hand-edit, the passes generalize to other kernels.

Why HGMMA needs the full round trip

HMMA and HGMMA aren’t two spellings of one instruction — they’re different execution models, so “use HGMMA” can’t be a textual swap. Everything feeding the multiply changes with it:

AspectHMMA (sm_80-class)HGMMA · wgmma (sm_90a)
Scope1 warp · 32 threadswarpgroup · 4 warps / 128 threads
Operandsregisters (LDSM from smem)shared-memory matrix descriptors
Syncwarp-synchronousasync + mbarrier
Accumulatorwarp register layoutwarpgroup register layout

So the rewrite goes through MLIR. At the SASS level the matmul is fused into registers, schedule and sync — it can’t be safely edited in place. Lifted to MLIR it becomes a single typed op that a conversion pass can legally re-express in Hopper’s warpgroup form, then lower back to sm_90a SASS and reinject.

Show the MLIR detail
  1. Lift SASS → MLIR. Raise the machine code into a typed IR that captures semantics: “an M×N×K fp16 multiply-accumulate,” “a global→shared copy,” the loop/region structure — abstracting away physical registers and schedule. The unusual direction: compilers lower down; sass2mlir raises up.
  2. Conversion pass. Pattern-match the matmul op and re-express it in the warpgroup paradigm: per-warp mma → wgmma; operands re-sourced from shared-memory descriptors; loads converted to TMA; an mbarrier async pipeline inserted; work split producer/consumer across the warpgroup; the accumulator moved to the wgmma layout. Match → replace → legalize — a structured transform, not a patch. The spill-free uniform-register path falls out of the leaner Hopper sequence.
  3. Re-encode + reinject. Lower back to SASS targeting sm_90a (so HGMMA/ TMA can actually be emitted) and patch it into the cubin’s .text (+ updated .nv.info/reloc), preserving the kernel ABI.

Injecting the recompiled cubin

The end product is a single self-contained cubin. To prove it inside an unmodified flash_attn process — without rebuilding PyTorch — we use an LD_PRELOAD harness, but it does something fundamentally simpler than the runtime swap: it replaces bytes, not an ABI. The recompiled cubin loads into the official kernel slot, launches with the right thread/shared-memory config, and produces correct output — same numbers as the official kernel, faster.

How the byte-level injection works

The hard part is where the cubin enters the process. flash_attn doesn’t hand the loader a raw cubin — it hands it a fatbin (a container of cubins for several architectures). The driver picks the matching arch and extracts the raw ELF cubin into a fresh buffer with memcpy. So the injection point isn’t a CUDA call at all — it’s that copy:

  1. Interpose memcpy. Unlike driver entry points, memcpy is reachable by ordinary LD_PRELOAD symbol interposition. The shim forwards every copy to the real memcpy, then inspects the destination: if it now holds a resident ELF cubin whose section table carries the target kernel’s mangled symbol, it overwrites that buffer in place with our recompiled cubin (header already carrying the sm_90a e_flag, plus the rewritten .text/.nv.info). The whole-cubin variant catches the 3.14 MB extraction in one shot; a section-overlay variant patches only .text/.nv.info when the layout matches. Bounds are gated by the copy length, so a partial or false ELF match finds nothing rather than reading wild memory.
  2. Two launch fixups. The recompiled kernel is a 256-thread warpgroup kernel that needs ~98 KB of dynamic shared memory. Via the resolver-hooked cuModuleGetFunction we grab the target handle; via cuLaunchKernel(Ex) we set the block to 256 and raise MAX_DYNAMIC_SHARED_SIZE to cover the smem footprint.
  3. Forward the parameters verbatim. This is the crux: the Flash_fwd_params buffer is passed unchanged. No marshalling, no tensormap rebuild, no pointer translation — because the recompiled kernel keeps the official ABI. It reads q/k/v/o and the strides from the exact same const-bank offsets the official kernel did; it just builds its TMA descriptors in-kernel from those same fields.

That last point is the whole argument for doing this in the compiler instead of at launch. The runtime swap has to translate the ABI on every call — read the official params, rebuild a foreign 34-argument buffer and four tensormaps, then launch. The offline reinject translates nothing: it swaps the machine code once, the ABI is identical, and the launch site is untouched. That’s why it avoids the swap’s small-sequence penalty entirely — and why it ships as one binary instead of two.

The IR

sass2mlir: an MLIR dialect for machine code

Underneath the round trip is a real MLIR dialect for NVIDIA SASS. It lifts raw GPU machine code into custom SSA ops, lets you transform it as a real IR, and lowers it back to byte-exact bytes — so a compiled kernel’s .text can be rewritten and patched into a cubin with no source recompile. Each SASS instruction becomes one op named for its mnemonic and opcode (sass.<MNEMONIC>_0x<opcode>), carrying its SSA operands and results, an inline guard(%p) predicate, modifiers glued to the name, and the authoritative 128-bit raw encoding that makes lowering exact.

Because lift→lower is byte-exact, any difference in the output is exactly the transform you applied — nothing else moves. That’s what makes it safe to optimize or retarget a shipped kernel at the machine-code level and drop the result straight back into the original cubin.

Show a real lifted Hopper (sm_90a) kernel

Control flow is recovered into MLIR basic blocks; φ-nodes at joins become block arguments (Braun SSA), so cross-block dataflow is native MLIR.

module {
  sass.func "lifted" {
    %0 = builtin.unrealized_conversion_cast to !sass.reg     // kernel live-in values
    ...
    // fused multiply-add: predicated, negated first operand, one result
    %4706 = sass.FFMA_0x223.FTZ guard(%4380) -%4550, %4464, %4551
              : (!sass.reg, !sass.reg, !sass.reg) -> (!sass.reg)

    // warpgroup matrix-multiply: two results, four operands
    %5717:2 = sass.HGMMA_0xdf0.64x8x16.F32 guard(%425) %5716, %5590, %5716, %5716
              : (!sass.reg, !sass.reg, !sass.reg, !sass.reg) -> (!sass.reg, !sass.reg)

    sass.UTMALDG_0x5b4.3D guard(%266) %31250, %31086 : (!sass.reg, !sass.reg) -> ()  // TMA load
    sass.STSM_0x844.16.M88.4 %33430, %33553 : (!sass.reg, !sass.reg) -> ()           // smem store

    cf.br ^bb1(%4783, %4751, ... : !sass.reg, ...)   // block args are SSA phis
    sass.exit
  }
}

The SSA values are the analysis view that decides what to change; raw is the ground truth that gets rewritten and lowered. The conversion above (c[0x0][0x210] → c[0x0][0x2e8]) is exactly the decode → match → re-encode → mask-verify loop in the widget: it edits a single field and proves, bit by bit, that nothing else moved.

Alternative approach

The runtime swap: when a rewrite outgrows its slot

There’s more than one way to get optimized machine code into a running process, and we built a second one: a runtime swap — an LD_PRELOAD shim that hooks the CUDA driver’s symbol resolver, intercepts the cuLaunchKernelEx call for the official kernel, and relaunches the optimized Triton kernel from its cubin (templating its parameter buffer and patching only the device pointers). It lifts the official call site from 102 to 132 runs/s.

This is a genuinely useful technique — just not the one this flash-attention rewrite needs. Offline reinjection patches the new code into the cubin’s existing .text slot, which works as long as the rewrite fits. The moment an optimization makes a function grow past the space allotted to it in the binary, you can’t patch in place — there’s nowhere to put the extra bytes. A runtime swap sidesteps that entirely: it launches a separately compiled kernel at the original call site, so the replacement’s size is unconstrained by the original’s footprint. Our flash-attention rewrite fits its slot, so reinjection is the right tool here — we built the swap to demonstrate the capability a different way, and as the approach you’d reach for when a rewrite doesn’t fit.

How the runtime swap actually hooks the driver

You can’t just export your own cuLaunchKernelEx and let LD_PRELOAD win. The CUDA runtime doesn’t call driver entry points by their linker symbol — it resolves them dynamically at runtime through dlsym and the driver’s own cuGetProcAddress. A plain symbol interposition is simply never reached. So the shim hooks the resolvers:

  1. Bootstrap the real dlsym. Our dlsym override needs the genuine one to forward to, but asking dlsym for dlsym is circular. The escape hatch is the versioned lookup: dlvsym(RTLD_NEXT, "dlsym", "GLIBC_2.2.5") hands back libc’s real dlsym without going through the symbol we’re overriding.
  2. Intercept both resolvers. Our dlsym and cuGetProcAddress watch for requests for cuLaunchKernelEx (the entry Triton and modern CUDA launch through) and return a pointer to our function instead of the driver’s. Every launch in the process now flows through us.
  3. Identify and divert the official launch. At each call we check the kernel function handle. For anything but the official flash_fwd kernel we forward to the real driver untouched. For the official kernel we don’t launch it — we marshal a fresh parameter buffer for the Triton kernel: read the device pointers (q/k/v/o) and shapes out of the official 464-byte Flash_fwd_params, build the four TMA tensormap descriptors Triton expects, and launch the Triton cubin’s kernel with its grid/block/dynamic-smem config.

That per-launch marshalling — a 34-argument buffer plus four tensormaps rebuilt every call — is exactly the fixed cost that makes short sequences regress in the sweep below.

The trade-off is that it ships two kernels and pays an ABI translation on every launch (the parameter marshalling above), rather than producing a single self-contained binary the way reinjection does. Different tool, different job.

Both compute (51% → 40.97%) and memory (51.58% → 22.49%) throughput drop — yet latency drops too. Throughput-% measures hardware pressure, not speed. The rewrite does the same attention with less wasted work (spills gone → less memory traffic; HGMMA + uniform registers → fewer instructions issued), so it finishes faster while leaning on the hardware less.

A sequence-length sweep (10k iterations each, H100) shows the win grow with context length. At short sequences (≤1k tokens) the Triton path is actually a few percent slower — not because of the math, but because the Triton runtime wraps its kernel variables heavily: each launch marshals a 34-argument parameter buffer including four TMA tensormap descriptors (vs the official kernel’s single 464-byte Flash_fwd_params struct), on top of warp-specialization and mbarrier setup. That fixed per-launch cost dominates when there’s barely any attention work to do. From ~4k tokens up the kernel win takes over, settling at 1.26–1.29× — the long-context regime that matters. Status

Where this stands

Working today: the complete pipeline — SASS disassembly (~100%), MLIR lift, the conversion passes (the six rules), SASS re-encode, and offline reinjection: the recompiled .text and updated .nv.info/reloc are written back into the cubin to ship a single self-contained, optimized kernel that drops into the official slot with no host changes and no runtime swap (official ABI, correct output). The runtime swap remains as the separate technique for rewrites that outgrow their slot (132 runs/s). The method itself — including an online, on-the-fly conversion path that rewrites kernels as they load — is the subject of a paper currently in peer-review prep, which is why the implementation details and full validation set are held back here.

Next: broaden the conversion-pass library beyond the six rules so the lift→transform→ re-encode path generalizes to more kernels and more optimization patterns. As those passes get more aggressive, the rewritten code will grow past what fits in the original .text slot — which is exactly the case the runtime swap was built for. The two techniques are complementary: reinjection while the rewrite fits, swap once it outgrows its footprint.
